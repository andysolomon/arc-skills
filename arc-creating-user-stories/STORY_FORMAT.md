# Story Format Reference

Load this when drafting story bodies or reviewing acceptance criteria. This file is the canonical work-item format for the arc skill family; `arc-defining-work`, `arc-linear-issue-creator`, and destination references point here.

## Story numbering

Every story uses a sequential zero-padded ID:

- `W-000001`, `W-000002`, ...

The ID is prepended to the issue title:

```text
[W-000001] Replace placeholder weather values
```

Before assigning IDs, check the highest existing story number in the destination tracker:

```bash
# GitHub Issues
gh issue list --limit 100 --state all --json title --jq '.[].title' | grep -oP 'W-\d+' | sort -r | head -1
```

For Linear, query issue titles for `W-\d{6}` (see `arc-linear-issue-creator`). For Agile Accelerator, `Name` is auto-generated by Salesforce; never set it during creation.

If no stories exist, start at `W-000001`; otherwise increment from the highest.

## Story body template

```markdown
## User Story
**ID:** W-000001

When [situation], I want [motivation], so that [outcome].

## Acceptance Criteria

- [ ] [Observable outcome stated as a checkable fact]
  - Verify: [exact test command, CLI check, or manual observation]
- [ ] [Another observable outcome]
  - Verify: [how an implementer proves this criterion]

## Context
Brief technical context referencing specific files, APIs, behaviors, or design constraints.
```

The story sentence is a **job story**: lead with the triggering situation, not a persona. "When the dashboard loads on a 375px viewport" carries more implementation signal than "As a mobile user".

## Acceptance-criteria rules

- Each criterion is one checkbox stating one observable outcome.
- Every criterion has a `Verify:` line naming a concrete method: a test to run or write, a command whose output proves the outcome, or a specific manual observation.
- A criterion an implementing agent cannot check is not done being written.
- Prefer concrete values ("responds within 200ms", "returns 403") over vague language ("fast", "secure").
- Cover the unhappy path when the story has one: at least one criterion for failure, empty, or permission-denied behavior.

## Labels / metadata

- Epic: `epic: <name>`
- Priority: `P0`, `P1`, `P2`
- Size: `S`, `M`, `L`, `XL`
- Type: `AFK` (agent-implementable end to end) or `HITL` (needs human action such as keys, accounts, or approvals)

## Quality rules

- Keep stories independently deliverable and small enough for one agent or developer.
- Prefer concrete context over generic feature descriptions.
- Mark dependencies (`Depends on: W-XXXXXX`) and blocked/HITL status explicitly.

## Example

```markdown
## User Story
**ID:** W-000015

When I open a city's weather panel on a phone, I want the panel and charts to fit the screen, so that I can browse without fighting the layout.

## Acceptance Criteria

- [ ] At viewport widths of 375px or less, the weather panel spans the full viewport height with the close button in the bottom half of the screen
  - Verify: Playwright test at 375x667 asserts panel height equals viewport height and close-button y-position > 50% of viewport
- [ ] Charts stack in a single column on mobile with no horizontal scrollbar
  - Verify: Playwright test asserts `document.documentElement.scrollWidth <= window.innerWidth` after charts render
- [ ] Desktop layout (>= 1024px) is unchanged
  - Verify: existing desktop visual-regression snapshots pass unmodified

## Context
Panel component: `src/components/WeatherPanel.tsx`
```
